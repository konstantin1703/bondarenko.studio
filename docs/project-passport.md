## 5. Важные принятые решения

### 5.1 CTA-система

Основной CTA по сайту:

> Обсудить проект

Поведение основного CTA:

- открывает модальное окно заявки;
- не ведёт напрямую в Telegram;
- не ведёт на email;
- не ведёт на телефон.

Прямые контакты разрешены только:

- в футере;
- внутри модального окна, где есть Telegram.

Запрещено размещать прямые Telegram/email/phone CTA в:

- hero;
- pre-footer;
- карточках услуг;
- карточках проектов;
- промежуточных CTA-блоках.

### 5.2 Pre-footer

Pre-footer должен быть без прямых контактов.

Текущая логика:

- заголовок: `Есть проект? Разберём задачу.`;
- текст объясняет, что можно описать задачу: сайт, лендинг, текст, упаковка или аудит;
- primary: `Обсудить проект` → модалка;
- secondary: `Смотреть услуги` → `/services/`.

### 5.3 Telegram

Telegram остаётся:

- в футере;
- внутри модалки.

Telegram не должен быть главным CTA на страницах.

### 5.4 Email и телефон

Email и телефон скрыты от простых HTML-парсеров через `data-*` атрибуты и восстанавливаются через vanilla JS.

Обычный пользователь видит и кликает:

- `mailto:` для email;
- `tel:` для телефона.

В исходном HTML не должно быть открытого `mailto:byxapckuu@gmail.com` и открытого номера телефона.

### 5.5 Мобильное меню

Мобильное меню уже добавлено.

Текущая логика:

- burger слева перед логотипом;
- EN и переключатель темы справа;
- desktop CTA скрывается на малых экранах;
- CTA `Обсудить проект` есть внутри мобильного меню;
- меню закрывается по ссылке, Escape и клику вне меню.

### 5.6 Футер

На внутренних RU-страницах футер добавляется через общий JS, если в HTML нет собственного `<footer>`.

Футер содержит:

- Telegram;
- email;
- телефон;
- нижнюю навигацию;
- copyright.

Главная страница имеет собственный футер и не должна получать дубль футера.

Если новый чат начинается с этого паспорта, ассистент должен:

1. Прочитать паспорт.
2. Не начинать аудит с нуля.
3. Считать текущий сайт рабочим.
4. Продолжить с ближайшей задачи: финальный OG PNG и замена `og:image` / `twitter:image`.
5. Работать через GitHub-коннектор, если нужно менять репозиторий.
6. После правок проверять GitHub Actions.
7. Не предлагать лишнюю перестройку сайта без причины.
